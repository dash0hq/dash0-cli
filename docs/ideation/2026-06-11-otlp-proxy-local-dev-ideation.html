Codebase Context

The dash0-cli is a Go CLI built on cobra, domain-organized under internal/. Two send commands exist today: logs send (internal/logging/send.go, ~220 lines) and spans send (internal/tracing/spans_send.go, ~327 lines). Both are one-shot, flag-driven; logs send takes a single <body> positional plus repeatable --resource-attribute / --log-attribute / --scope-attribute. The OTLP client comes from dash0-api-client-go and exposes SendLogs / SendTraces / SendMetrics over HTTP/JSON only — no gRPC.

No proxy, tail, follow, fsnotify, stdin-reading, or multi-signal accept patterns exist yet. Shared OTLP utilities live in internal/otlp/ (key-value parsing, scope defaults, trace/span ID parsing). Config resolution follows env > flags > profile, with the CLI distinguishing api-url from otlp-url. The existing api command (internal/rawapi/) sets the precedent for experimental escape-hatch commands behind -X: it reuses the active profile for connection, owns the Authorization header (rejecting it as -H input), and is documented as a stop-gap until first-class subcommands ship.

Novel command shapes must be gated -X initially per the project's adding-commands guide; the apply-parity rule applies only to Asset CRUD and exempts both proposals. Agent-mode constraints (NDJSON output, no colored output, no confirmation prompts) apply to every command, but the project has not yet made a design choice on NDJSON heartbeats for long-running commands.

External context: no major vendor ships a "proxy" subcommand in their CLI today. Datadog requires its full agent; Honeycomb / New Relic / Grafana require Collector configs. The OTel 2026 community survey reports 63 percent of Collector users cite config management as their top pain — minimum bearer-token forwarding is roughly 25 lines of YAML across four concepts (receivers, exporters, extensions, pipelines). otel-desktop-viewer (824 stars) confirms appetite for local OTLP inspection but ships no forwarder. Log-tail UX has converged across Fluent Bit, Vector, and Promtail on glob paths, SQLite/disk checkpoints, explicit start-from-beginning-vs-tail control, and inline multiline config; OTel's own filelog-receiver is the documented complexity outlier (issues #35162 and #31256 catalog multiline misconfiguration and lack of harvest observability). Cross-domain analogies that match the shape: kubectl proxy, ngrok, mkcert, gcloud auth application-default login, aws-vault exec, direnv.

Topic Axes

Five orthogonal surfaces the survivor set spreads across.

  1. Receive surface — what the CLI accepts as input (OTLP HTTP, OTLP gRPC translation, files, stdin/pipes, structured streams, env-var auto-injection into subprocesses).
  2. Auth & connection management — bearer token injection, profile-aware credentials, multi-dataset routing, header policy on inbound vs outbound, multi-tenant local routing.
  3. Local developer ergonomics — port selection, env-var propagation to subprocesses, integration with OTEL_EXPORTER_OTLP_* standards, lifecycle (Ctrl-C, daemonize, autostart), directory-aware behavior.
  4. Buffering, batching, & failure modes — offline behavior, backpressure, batching defaults, retry, restart-resilience, file checkpoints, rotation handling.
  5. Visibility & feedback — how the user knows data is flowing (logging, NDJSON heartbeats, optional local preview, byte/record counts, error surfacing, in-flight diagnostics).

I2

Zero-config dash0 otlp proxy: no flags that matter

Axis: Local developer ergonomics Confidence: High (80%) Complexity: Medium
Description

dash0 otlp proxy with no arguments picks an OS-assigned port (default --port 0), exposes the OTLP/HTTP spec paths (/v1/logs, /v1/traces, /v1/metrics) on one listener, uses the active profile's auth and dataset, prints exactly two lines on start (the OTEL_EXPORTER_OTLP_ENDPOINT to export, plus a live forwarded-records counter), and exits cleanly on Ctrl-C. Every flag exists, but every flag has a sensible default. No receivers, exporters, extensions, or pipelines concepts.

Basis

externalmkcert ("collapse 10 manual steps into one"), kubectl proxy (exact structural match: local HTTP server, CLI handles auth/TLS, Ctrl-C to stop), OTel 2026 community survey reporting 63 percent of Collector users cite config management as top pain. The minimum bearer-token forwarding Collector config is ~25 lines of YAML across four distinct concepts; a Dash0-only proxy collapses that to a spec-defined URL layout. Three convergent frame agents.

Rationale

The user's stated goal is "simplify"; this is the operational definition. The mkcert framing applies directly: a single command collapsing what would otherwise require multi-concept YAML. The OTel Collector exists for cases the Dash0 proxy explicitly does not address (any-backend, any-receiver, complex pipelines); a Dash0-only proxy can be radically simpler because its target is fixed. Pairs naturally with idea I1 (subprocess wrap) — they can be the same code path with different invocation modes.

Downsides

"Zero config" is achievable only if the active profile is already set up. New users hit the wall at profile creation; this composes with an OAuth-on-first-run path that the in-flight feat/oauth-login branch is preparing. Hides a useful escape hatch — power users wanting custom batching, gRPC ingest, non-default scope defaults, or different OTLP listener paths need flags that exist but aren't surfaced. Discoverability of the escape hatch is a UX problem the docs need to solve.

I3

Pipe-first dash0 logs tail: files become explicit opt-in

Axis: Receive surface Confidence: High (80%) Complexity: Medium
Description

Invert the filelog framing. The default surface is the pipe: myapp 2>&1 | dash0 logs tail - reads stdin line-by-line, auto-detects JSON-lines (fields lift to attributes), logfmt, or plain text (becomes the log body). Files are the explicit opt-in: dash0 logs tail '/var/log/myapp/*.log'. When tailing files, ship the converged Fluent Bit / Vector / Promtail UX — SQLite checkpoint at ~/.dash0/tail-state/, CRC fingerprinting for rotation survival, default --from-beginning (opposite of filelog-receiver's surprising start_at: end), single --multiline-start "<regex>" flag instead of the receiver's separate-parser design. Ship as a new subcommand logs tail, not a flag on logs send — preserves logs send's one-shot contract that the send-log-event GitHub Action depends on.

Basis

externalFluent Bit's tail input (Path, Read_from_Head, DB SQLite checkpoint, Multiline.Parser), Vector's file source (CRC fingerprinting, read_from: end, inline multiline), Promtail's pipeline_stages. OTel filelog-receiver pain points cataloged in issues #35162 and #31256 — multiline misconfiguration described as "most mysterious log parser," no observability into files being harvested, start_at: end default surprises new users. Three convergent frame agents.

Rationale

Most local-dev "I want my logs in Dash0" cases are actually "I want stdout/stderr of npm run dev in Dash0." A pipe handles that without ever introducing the file — shells already solve "follow a process's output." Picking up the convergent Fluent Bit / Vector / Promtail UX while explicitly diverging from filelog-receiver's complexity hits the user's "simplify" framing on the head. The new subcommand naming resolves verb confusion (logs send is one-shot; logs tail is long-running) and keeps logs send's GitHub Action contract intact.

Downsides

Two ways to do "logs from a running process" (pipe or file) — readers need to learn the distinction. CRC fingerprinting and SQLite checkpoints add real implementation surface (Vector's took years to get right). Auto-detection of structured-vs-plain has edge cases: JSON-lines with stray non-JSON lines, logfmt with embedded equals signs, mixed content streams. Needs an explicit --format json|logfmt|plain opt-out.

Open Questions for Brainstorm

Things the chosen direction (I2 + I3) leaves open. The brainstorm pass should commit one position on each — the ideation doc surfaces them, the brainstorm doc resolves them.

CC1 — Lifecycle: daemon-only, or compose with a subprocess wrapper later?

I2 ships as a long-running daemon (Ctrl-C to stop). A future dash0 exec -- <cmd> wrapper (deferred at Phase 5) could share the same listener implementation and inject env vars into a child. The brainstorm should decide whether to design I2's internals so that exec mode plugs in later without rework, or commit to daemon-only and revisit when the demand surfaces.

CC2 — Auth and dataset routing: broker pattern vs passthrough

Both I2 and I3 need to know where the auth token comes from. Two viable models: (a) credential broker — inbound carries no auth, CLI injects bearer from the active profile on the outbound hop (matches the existing rawapi precedent); (b) passthrough — apps set their own headers, proxy just forwards. Per-request dataset routing (via header or resource attribute) is a separate optional layer on top of either model.

CC3 — Buffering policy on Dash0 unreachable

Three positions to choose between: durable WAL-backed disk spool (survives laptop sleep, big implementation surface), fail-loud (return 503, cuts implementation in half, surfaces problems immediately), or a small in-memory ring buffer (a middle ground — survives transient blips, no disk state). The pipe-first tail (I3) has the same question — what happens to lines that arrive while offline?

CC4 — Visibility default: silent, chatty, or NDJSON-by-default

The proxy and tail are both long-running commands without a precedent in the CLI. The Phase 5-deferred I5 explored one direction (deep-links per record + NDJSON heartbeats + optional inspect TUI). Brainstorm needs to pick a sensible default for TTY vs agent mode, and decide whether to lock the NDJSON event schema now (so a future exec or wrap command inherits it for free) or wait.

CC5 — Command placement: under dash0 otlp + dash0 logs or a future dash0 dev umbrella

Today's user prompt naturally lands at dash0 otlp proxy and dash0 logs tail. A future dash0 dev umbrella was considered and deferred at Phase 5. The brainstorm should commit to the v1 placement (the obvious one) and note whether a later move to dev would be a rename or a parallel surface.
